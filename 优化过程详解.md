## 整体流程概述

### 用户完整业务流程

```
1. 获取待拣货列表
   ↓
2. 按 UPC 每 15 份整理（此时不知道拣货单信息）
   ↓
3. 发送请求获取拣货单数据
   ↓
4. 存回拣货单信息
   ↓
5. 运行 optimize-pick-batches.js 进行优化
   ↓
6. 生成 PDF
```

### 优化脚本核心目标

- **single 类型**：以拣货单为最高优先级，先按 zone 分，再按 zone 对应的 shelfName 来
- **multiple 类型**：考虑组合绑定关系（A+B、B+C），使用启发式聚类
- **两者共同**：每批最多 15 个 UPC，尽量减少 zone/shelf 切换

## 问题发现与分析

### 问题 1：货位排序混乱

**现象**：
- CC 货区后面紧跟着 HH 货区
- 从优化算法角度来看这种情况不应该发生

**初步分析**：
- 初始算法过于激进，完全禁止了不同货区前缀的混合
- 没有考虑到数量控制的需求（每批需要尽可能接近 15 个 UPC）

### 问题 2：single 类型中 zone 混乱

**现象**：
- "2200 还没排完就塞了其他的 zone"
- 同一批次中混合了多个 zone

**深入分析**：
- 怀疑是单个 UPC 组本身就包含了多个 zone
- 创建 `analyze-groups.js` 进行验证

**分析结果**：
```
single 类型:
  总共有 284 个 group
  包含多个 zone 的 group 数: 2 (0.70%)

multiple 类型:
  总共有 24 个 group
  包含多个 zone 的 group 数: 4 (16.67%)
```

**结论**：
- 这是原始数据问题（仓库库存分布）
- 无法在优化阶段彻底解决

## single 类型优化详解

### 优化前的问题

**启发式聚类的问题**：
```
假设仓库路径是：AA001 → AA002 → BB001 → BB002 → CC001 → CC002

启发式聚类可能会这样：
  种子：CC001
  下一个：CC002（同 shelf，高分）
  下一个：BB002（距离近，高分）
  下一个：BB001（同 shelf）
  ...

结果：拣货员从 CC 走到 BB，再走回 CC，路径回退
```

### 优化后：全局路径排序

**核心思路**：
1. 把所有 group 按仓库路径顺序排列
2. 顺序切分成每批最多 15 个 UPC

**排序函数**：`compareSingleGroupsByPath`

```javascript
function compareSingleGroupsByPath(a, b) {
  // 1. 优先按 zone 排序
  if (a.location.dominantZone !== b.location.dominantZone) {
    return compareZones(a.location.dominantZone, b.location.dominantZone);
  }
  
  // 2. 再按 shelf 排序
  if (a.location.dominantShelf !== b.location.dominantShelf) {
    return String(a.location.dominantShelf).localeCompare(
      String(b.location.dominantShelf)
    );
  }
  
  // 3. 最后按锚点位置排序
  if (a.location.anchorPosition.row !== b.location.anchorPosition.row) {
    return a.location.anchorPosition.row - b.location.anchorPosition.row;
  }
  if (a.location.anchorPosition.column !== b.location.anchorPosition.column) {
    return a.location.anchorPosition.column - b.location.anchorPosition.column;
  }
  if (a.location.anchorPosition.side !== b.location.anchorPosition.side) {
    return a.location.anchorPosition.side - b.location.anchorPosition.side;
  }
  
  return 0;
}
```

**切分函数**：`optimizeSingleBatchesByPath`

```javascript
function optimizeSingleBatchesByPath(batches, boxSize) {
  // 1. 全局排序
  const groups = buildGroups("single", batches).sort(compareSingleGroupsByPath);
  
  // 2. 顺序切分
  const optimizedBatches = [];
  let currentGroups = [];
  let currentUsedSlots = 0;
  
  for (const group of groups) {
    const nextUsedSlots = currentUsedSlots + group.slotCost;
    
    // 超过箱容，刷新当前批次
    if (currentGroups.length > 0 && nextUsedSlots > boxSize) {
      flushCurrentBatch();
    }
    
    currentGroups.push(group);
    currentUsedSlots += group.slotCost;
  }
  
  return optimizedBatches;
}
```

**优势**：
- ✅ 路径最优：从仓库头走到尾，不回退
- ✅ 计算简单：O(n log n) 排序 + O(n) 切分
- ✅ zone 切换最少

## multiple 类型优化详解

### 为什么 multiple 不用全局排序？

**multiple 的特点**：
- group 之间有组合绑定关系（A+B、B+C 这种组合单）
- 需要尽量保持组合在一起
- 如果硬切，会破坏组合关系

### 启发式聚类算法流程

```
1. 把所有 group 按路径排序（选择种子用）
   ↓
2. 从剩余 group 中选第一个作为种子
   ↓
3. 从剩余 group 中找出与当前批次最匹配的 candidate
   ↓
4. 如果 candidate 能放下，加入批次；否则刷新当前批次
   ↓
5. 重复 2-4，直到所有 group 都处理完
```

### 种子选择逻辑

**排序函数**：`compareGroupsByAnchor`

```javascript
function compareGroupsByAnchor(a, b) {
  // 1. zone 靠前优先
  if (a.location.dominantZone !== b.location.dominantZone) {
    return compareZones(a.location.dominantZone, b.location.dominantZone);
  }
  
  // 2. 行靠前优先
  if (a.location.anchorPosition.row !== b.location.anchorPosition.row) {
    return a.location.anchorPosition.row - b.location.anchorPosition.row;
  }
  
  // 3. 列靠前优先
  if (a.location.anchorPosition.column !== b.location.anchorPosition.column) {
    return a.location.anchorPosition.column - b.location.anchorPosition.column;
  }
  
  // 4. 侧靠前优先
  if (a.location.anchorPosition.side !== b.location.anchorPosition.side) {
    return a.location.anchorPosition.side - b.location.anchorPosition.side;
  }
  
  // 5. 件数多的优先
  if (b.location.totalQty !== a.location.totalQty) {
    return b.location.totalQty - a.location.totalQty;
  }
  
  // 6. 最后按 UPC
  return String(a.upcUnique).localeCompare(String(b.upcUnique));
}
```

**结论**：优先选择库位靠前的 group 作为种子

### 评分函数详解

**函数**：`scoreCandidate`

```javascript
function scoreCandidate(candidate, context, boxSize) {
  const nextUsedSlots = calcCandidateNextUsedSlots(context, candidate);
  
  // 放不下，负无穷
  if (nextUsedSlots > boxSize) {
    return -Infinity;
  }
  
  const addedSlots = nextUsedSlots - context.usedSlots;
  
  // 第一个 group，种子评分
  if (context.groups.length === 0) {
    return (
      candidate.location.totalQty * 10 -
      candidate.location.anchorPosition.travelIndex -
      addedSlots
    );
  }
  
  // 计算各种共享
  const sharedZones = calcSetIntersectionSize(candidate.location.zoneSet, context.zoneSet);
  const sharedShelves = calcSetIntersectionSize(candidate.location.shelfSet, context.shelfSet);
  const sharedPrefixes = calcSetIntersectionSize(candidate.location.prefixSet, context.prefixSet);
  const distance = Math.abs(candidate.location.avgTravelIndex - context.avgTravelIndex);
  const remains = boxSize - nextUsedSlots;
  
  let score = 0;
  
  // 共享 shelf 权重最高
  score += sharedShelves * 4000;
  
  // 共享货区前缀（CC/BB）权重次高
  score += sharedPrefixes * 10000;  // 从 5000 提升到 10000
  
  // 共享 zone
  score += sharedZones * 900;
  
  // 主导 shelf 在当前批次中
  if (context.shelfSet.has(candidate.location.dominantShelf)) {
    score += 3000;
  }
  
  // 主导 zone 在当前批次中
  if (context.zoneSet.has(candidate.location.dominantZone)) {
    score += 1800;
  }
  
  // ✨ 2026-04-07 新增：无共享前缀时加大惩罚
  if (sharedPrefixes === 0) {
    score -= 10000;
  }
  
  // 距离惩罚
  score += Math.max(0, 1200 - distance * 15);
  
  // 行号惩罚
  score += Math.max(0, 600 - candidate.location.anchorPosition.row * 20);
  
  // 剩余槽位奖励
  score += Math.max(0, 150 - remains * 10);
  
  // 新增槽位惩罚
  score += Math.max(0, 300 - addedSlots * 20);
  
  // 数量奖励
  score += candidate.location.totalQty * 5;
  
  return score;
}
```

**评分权重说明**：
| 因素 | 权重 | 说明 |
|------|------|------|
| sharedShelves | 4000 | 同一货架最重要 |
| sharedPrefixes | 10000 | 同一货区前缀（CC/BB）次重要 |
| 无 sharedPrefixes 惩罚 | -10000 | 尽量避免不同前缀混合 |
| sharedZones | 900 | 同一 zone |
| 距离惩罚 | 1200 - 15×距离 | 距离越远扣分越多 |

## UPC 大小写敏感问题

### 问题发现

**现象**：
- 原始数据中订单 UPC 是 "Ody-Marshmallow"（大写 M）
- 拣货单 labelDesc 中 UPC 是 "Ody-marshmallow"（小写 m）
- 因为大小写不同导致匹配失败，拣货单缺失

**深入分析**：
- 问题出在 `allocateLabelItemsByDemand` 函数
- 第 351 行使用 `if (itemUpc !== upc)` 严格比较
- 导致 "Ody-Marshmallow" 和 "Ody-marshmallow" 无法匹配

### 库存 UPC 重复分析

为了评估修复的安全性，我们检查了 1856 个库存 UPC：

```
检查结果：
- 总 UPC 数：1856 个
- 完全相同的重复：286 个
- 大小写不同的重复：1 组（"Santal 33" vs "santal 33"）
```

**结论**：库存数据非常干净，只有 1 组大小写不同的重复，修复是安全的。

### 修复方案

**修改 `allocateLabelItemsByDemand` 函数**（第 351 行）：

```javascript
// 修改前
if (itemUpc !== upc) continue;

// 修改后
if (itemUpc.toLowerCase() !== upc.toLowerCase()) continue;
```

**验证结果**：
- 修复前：self.json 中 multiple 类型有 61 个 UPC 找到匹配
- 修复后：self.json 中 multiple 类型有 72 个 UPC 找到匹配
- 新增找到的 UPC 包括："Ody-Marshmallow"、"luminous pink"、"Rhubarb" 等

### single vs multiple 的处理差异

| 类型 | UPC 匹配方式 | labelItems 获取方式 | 受影响吗？ |
|------|-------------|-------------------|-----------|
| **single** | 不涉及 UPC 匹配 | 通过 orderNo 从 orderLabelMap 回收 | ❌ 不受影响 |
| **multiple** | 大小写不敏感匹配 | 通过 UPC 从 labelPool 分配 | ✅ 已修复 |

### 潜在问题分析

虽然我们修复了匹配问题，但还有一个潜在问题：

**聚合 key 问题**：`buildAggregatedOutputItemKey` 函数中的 UPC 仍区分大小写

```javascript
function buildAggregatedOutputItemKey(rawItem) {
  const item = normalizeLabelItem(rawItem, rawItem?.zoneName);
  return [
    item.zoneNo,
    item.zoneName,
    item.shelfNo,
    item.shelfName,
    item.upc,  // ← 这里用的是原样的 upc！没有转小写！
    item.sku,
  ].join("||");
}
```

**影响**：
- 如果同一库位中有 "santal 33" 和 "Santal 33"
- 它们会生成不同的聚合 key
- 在最终的拣货单中会是两个独立的 item
- 不会合并数量

**是否需要修复**：需要进一步评估业务需求

## 算法对比与选择

### 两种算法对比表

| 特性 | single（全局路径排序） | multiple（启发式聚类） |
|------|------------------------|------------------------|
| **适用场景** | group 之间独立，无绑定 | group 之间有组合绑定 |
| **排序方式** | 全局线性排序（类似蛇形路径） | 局部聚类（种子 + 最佳候选） |
| **路径连续性** | ✅ 最佳，从仓库头走到尾 | ⚠️ 一般，可能有回退 |
| **组合保持** | ❌ 不考虑 | ✅ 考虑 sharedShelves/sharedZones 等 |
| **实现复杂度** | 简单 | 复杂 |
| **计算效率** | O(n log n) | O(n²)（最坏情况） |
| **zone 切换** | ✅ 最少 | ⚠️ 稍多 |

### 为什么 single 用全局排序更优？

**single 的特点**：
1. group 之间独立，不需要考虑组合关系
2. 目标简单，就是沿仓库路径走一遍
3. 不需要局部聚类的灵活性

**全局排序的优势**：
```
仓库路径（示意）：
AA001 → AA002 → BB001 → BB002 → CC001 → CC002 → ... → HH001 → HH002

全局排序后的切分：
  Batch 1: AA001, AA002, BB001, BB002, ... (连续 15 个)
  Batch 2: CC001, CC002, ... (接下来 15 个)
  Batch 3: ...

结果：拣货员从 AA 走到 BB，再走到 CC，路径最优！
```

### 为什么 multiple 必须用启发式聚类？

**multiple 的特点**：
- group 之间有组合绑定关系（A+B、B+C）
- 需要尽量保持组合在一起

**举例**：
```
假设有组合单：
  Group A (CC001) + Group B (CC002)
  Group B (CC002) + Group C (CC003)

如果用全局排序 + 硬切：
  可能在 A+B 之间切，破坏组合关系

用启发式聚类：
  可以通过 sharedShelves/sharedPrefixes 评分来保持组合
```

// 从 "CC001A" 提取 "CC"
function extractPrefix(shelfName) {
  const match = String(shelfName).match(/^([A-Z]{2})/);
  return match ? match[1] : "";

### 3. 集合交集计算

```javascript
function calcSetIntersectionSize(setA, setB) {
  let count = 0;
  for (const item of setA) {
    if (setB.has(item)) {
      count++;
    }
  }
  return count;
}
```

### 4. Zone 比较

```javascript
function compareZones(a, b) {
  // 支持 "2200"、"2200-1" 等格式
  const numA = parseInt(String(a).split("-")[0], 10) || 0;
  const numB = parseInt(String(b).split("-")[0], 10) || 0;
  return numA - numB;
}
```

## 总结与思考

### 本次优化的核心改进

1. **single 类型**：从启发式聚类改为全局路径排序，路径最优
2. **评分函数**：加大货区前缀的权重（10000），无共享前缀时惩罚 10000
3. **问题定位**：发现单个 UPC 组本身跨 zone 的原始数据问题

### 仍存在的限制

- **原始数据问题**：单个 UPC 组跨 zone 的情况无法在优化阶段解决
- **multiple 的路径**：启发式聚类可能导致路径回退，这是 trade-off

### 未来可能的优化方向

1. **仓库路径建模**：根据实际仓库布局，建立更精确的路径模型
2. **动态权重调整**：根据不同场景（波次、时段）调整评分权重
3. **组合单识别**：更好地识别和处理 multiple 中的组合绑定关系

## orderNo 不可信的问题（2026-04-09 更新）

### 问题背景

在实际业务中发现，orderNo 已经不能作为可信的关联依据了。

**现象**：
- 当同一个 UPC 以不同大小写形式出现在 upcUniqueList 中时（如 "Shiliu-lv" 和 "shiliu-lv"）
- 根据订单号获取拣货单时，只会返回一个整合了的拣货单
- 这导致 single 类型也不能使用 orderNo 来做匹配了

**具体例子**：
```javascript
// upcUniqueList 中可能同时出现：
["Shiliu-lv", "shiliu-lv"]

// 但是根据订单号获取拣货单时：
// 只会返回一个整合了的拣货单，包含所有 Shiliu-lv 的拣货明细
// 不会区分大小写
```

### 影响范围

- **single 类型**：原本通过 orderNo 关联，现在也受影响
- **multiple 类型**：本来就不依赖 orderNo，通过 UPC 匹配

### 解决方案

我们采取了以下措施：

#### 1. single 类型也使用 outputItemKeySet
- 修改 `buildGroupOutputItemKeySet` 函数，移除类型判断
- single 类型也计算 outputItemKeySet
- 使用 `outputItemKeySet.size` 来计算 slotCost

#### 2. 统一使用 outputItemKey 聚合拣货单
- 修改 `buildAggregatedOutputItemKey` 函数
- 把 UPC 和 SKU 都转成小写来构建 key
- 确保大小写不同的相同 UPC 能被正确聚合

#### 3. single 类型也做拣货单聚合
- 修改 `aggregateOutputLabelItems` 函数，移除类型判断
- single 类型也做拣货单聚合

#### 4. UPC 大小写合并
- 新增 `preprocessBatch` 函数
- 合并大小写相同且 upcDescList 完全相同的 UPC
- 按 UPC 小写 + upcDescList 精确匹配分组
- 例如："AAA*3" 和 "AAA*1" 会被分到不同组，分别生成 cover

### single 类型批次优化

为了确保 single 类型批次尽量填满 15 个不同 SKU，我们做了以下优化：

#### 1. 先合并所有批次
- 新增 `mergeAllSingleBatches` 函数
- 把所有 single 批次合并成一个大批次
- 确保所有 SKU 都被充分利用

#### 2. 使用 outputItemKeySet 去重计算
- 修改 `optimizeSingleBatchesByPath` 函数
- 使用 `outputItemKeySet` 去重计算 slot 占用
- 确保每批尽量填满 15 个不同 SKU

#### 3. 移除特殊逻辑
- 移除了相同 UPC 且相同库位不切分的特殊逻辑
- 避免批次不均衡的问题

### 前后端同步

同时修改了 tianju-web-app 项目：

#### 1. 分组逻辑同步
- 修改 `src/utils/index.js` 中的 `extractAndGroupByUpcUnique` 函数

## 最新问题修复（2026-04-09）

### 1. Single 类型 labelDesc 数据缺失问题

**问题背景：**
- 之前的修复错误地将 Single 类型改用 `collectLabelItems`（基于 orderNo）
- 但实际业务中，多个只是大小写不同的 UPC（如 "Shiliu-lv" 和 "shiliu-lv"）会共享同一个 labelDesc item
- 导致一个 group 拿到数据，另一个 group 拿不到，出现 labelDesc 缺失问题

**具体例子：**
```javascript
// 原始数据
{
  upcUniqueList: ["Shiliu-lv", "shiliu-lv"],
  orders: [
    [{ orderNo: "105656820613", upcDescList: ["Shiliu-lv*1"] }],
    [{ orderNo: "105656824275", upcDescList: ["shiliu-lv*2"] }]
  ],
  labelDesc: {
    items: [
      { 
        upc: "Shiliu-lv", 
        num: 32,  // 这是两个 UPC 的总和！
        orderNo: "105656820613"  // 只是其中一个订单号
      }
    ]
  }
}
```

**影响范围：**
- Single 类型

**我们采取的做法：**
- Single 类型改回使用 `allocateLabelItemsByDemand`
- 不传 fallbackOrderNo，让它用原始 item 的 orderNo
- 因为拣货单不打印 orderNo，所以 orderNo 字段不重要

### 2. preprocessBatch 合并逻辑问题

**问题发现：**
- 之前的 preprocessBatch 用了 `collectLabelItems` 来找 labelItems
- 对于共享 labelDesc item 的 UPC 有问题，可能找不到 labelItems

**修复方案：**
- 直接从 batch.labelDesc 中查找与当前 UPC 小写匹配的 item
- 按 UPC 小写 + 库位信息作为合并 key
- 正确合并像 "Shiliu-lv" 和 "shiliu-lv" 这种共享 labelDesc item 的 UPC

### 3. labelDesc 中的 orderNo 不重要

**重要发现：**
- 查看 `process-ordersV4.js` 的 `generatePickList` 函数
- 拣货单只打印三列：UPC、Location、Qty
- **根本不打印 orderNo！**
- 所以 labelDesc 中的 orderNo 字段完全不重要

**结论：**
- orderNo 字段可以随便填或者留空
- 只要拣货单中的库位、UPC、数量正确就行
