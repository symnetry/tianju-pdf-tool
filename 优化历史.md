## 2026-04-09（最新更新）

### 核心问题分析

#### 1. Single 类型 labelDesc 数据缺失问题
**问题背景：**
- 之前的修复错误地将 Single 类型改用 `collectLabelItems`（基于 orderNo）
- 但实际业务中，多个只是大小写不同的 UPC（如 "Shiliu-lv" 和 "shiliu-lv"）会共享同一个 labelDesc item
- 导致一个 group 拿到数据，另一个 group 拿不到，出现 labelDesc 缺失问题

**具体例子：**
```javascript
// 原始数据
{
  upcUniqueList: ["Shiliu-lv", "shiliu-lv"],
  orders: [
    [{ orderNo: "105656820613", upcDescList: ["Shiliu-lv*1"] }],
    [{ orderNo: "105656824275", upcDescList: ["shiliu-lv*2"] }]
  ],
  labelDesc: {
    items: [
      { 
        upc: "Shiliu-lv", 
        num: 32,  // 这是两个 UPC 的总和！
        orderNo: "105656820613"  // 只是其中一个订单号
      }
    ]
  }
}
```

**影响范围：**
- Single 类型

**我们采取的做法：**
- Single 类型改回使用 `allocateLabelItemsByDemand`
- 不传 fallbackOrderNo，让它用原始 item 的 orderNo
- 因为拣货单不打印 orderNo，所以 orderNo 字段不重要

#### 2. preprocessBatch 合并逻辑问题
**问题发现：**
- 之前的 preprocessBatch 用了 `collectLabelItems` 来找 labelItems
- 对于共享 labelDesc item 的 UPC 有问题，可能找不到 labelItems

**修复方案：**
- 直接从 batch.labelDesc 中查找与当前 UPC 小写匹配的 item
- 按 UPC 小写 + 库位信息作为合并 key
- 正确合并像 "Shiliu-lv" 和 "shiliu-lv" 这种共享 labelDesc item 的 UPC

#### 3. labelDesc 中的 orderNo 不重要
**重要发现：**
- 查看 `process-ordersV4.js` 的 `generatePickList` 函数
- 拣货单只打印三列：UPC、Location、Qty
- **根本不打印 orderNo！**
- 所以 labelDesc 中的 orderNo 字段完全不重要

**结论：**
- orderNo 字段可以随便填或者留空
- 只要拣货单中的库位、UPC、数量正确就行

### `optimize-pick-batches.js` 修复

#### 1. Single 类型 labelDesc 获取方式修复
- **修改 `buildGroups` 函数**：
  - Single 类型改回使用 `allocateLabelItemsByDemand`
  - 不传 fallbackOrderNo，让它用原始 item 的 orderNo
  - 与 Multiple 类型保持一致的 labelItems 获取方式

#### 2. preprocessBatch 合并逻辑优化
- **修改 `preprocessBatch` 函数**：
  - 移除对 `collectLabelItems` 和 `orderLabelMap` 的依赖
  - 直接从 batch.labelDesc 中查找与当前 UPC 小写匹配的 item
  - 按 UPC 小写 + 库位信息（zoneNo + zoneName + shelfNo + shelfName）作为合并 key

### 验证结果
- ✅ Shiliu-lv 问题解决：两个 UPC 被正确合并
- ✅ labelDesc 数据完整：没有缺失
- ✅ 数据完整性检查通过：orderNo、orders、labelDesc 都没有缺失
- ✅ UPC 组数从 227 变成了 226：成功合并了一对 UPC
- ✅ 所有验证都通过

## 2026-04-09

### 核心问题分析

#### 1. orderNo 不可信的问题
**问题背景：**
- 当同一个 UPC 以不同大小写形式出现在 upcUniqueList 中时（如 "Shiliu-lv" 和 "shiliu-lv"）
- 根据订单号获取拣货单时，只会返回一个整合了的拣货单
- 所以 single 类型也不能使用 orderNo 来做匹配了，要学习 multiple 的做法

**影响范围：**
- single 类型和 multiple 类型都受影响

**我们采取的做法：**
- single 类型也使用 `outputItemKeySet` 来计算 slot 占用
- 使用 `outputItemKey` 来聚合拣货单（`buildAggregatedOutputItemKey` 函数）
- 把 UPC 和 SKU 都转成小写来构建 key

#### 2. single 类型批次没有填满 15 个不同 SKU 的问题
**问题发现：**
- 有些批次只有 11 个、12 个不同 SKU，没有填满 15 个
- 原因是 `optimizeSingleBatchesByPath` 函数只是简单地累加每个 group 的 slotCost，没有考虑到 `outputItemKeySet` 的去重
- 同时还保留了一个特殊逻辑：相同 UPC 且相同库位的 group 即使超过 15 个也不切分，导致批次不均衡

**修复方案：**
- 修改 `optimizeSingleBatchesByPath` 函数，使用 `outputItemKeySet` 去重计算 slot 占用
- 移除了相同 UPC 且相同库位不切分的特殊逻辑
- 确保每批尽量填满 15 个不同 SKU

#### 3. single 类型没有把所有批次先合并再处理的问题
**问题发现：**
- `preprocessBatch` 是对每个原始批次单独处理的
- 导致 single 类型的批次没有充分利用所有的 SKU

**修复方案：**
- 新增 `mergeAllSingleBatches` 函数，把所有 single 批次合并成一个大批次
- 修改 `optimizePickingData` 函数，在处理 single 类型前先合并所有批次

### `optimize-pick-batches.js` 重大改进

#### 1. UPC 大小写合并
- **新增 `preprocessBatch` 函数**：
  - 合并大小写相同且 upcDescList 完全相同的 UPC
  - 按 UPC 小写 + upcDescList 精确匹配分组
  - 例如："AAA*3" 和 "AAA*1" 会被分到不同组，分别生成 cover

#### 2. single 类型批次优化
- **新增 `mergeAllSingleBatches` 函数**：
  - 把所有 single 批次合并成一个大批次
  - 确保所有 SKU 都被充分利用

- **修改 `optimizeSingleBatchesByPath` 函数**：
  - 使用 `outputItemKeySet` 去重计算 slot 占用
  - 确保每批尽量填满 15 个不同 SKU
  - 移除了相同 UPC 且相同库位不切分的特殊逻辑

#### 3. 拣货单聚合优化
- **修改 `aggregateOutputLabelItems` 函数**：
  - 移除类型判断，single 类型也做拣货单聚合

- **修改 `buildAggregatedOutputItemKey` 函数**：
  - 把 UPC 和 SKU 都转成小写
  - 确保大小写不同的相同 UPC 能被正确聚合

- **修改 `buildGroupOutputItemKeySet` 函数**：
  - 移除类型判断，single 类型也计算 outputItemKeySet

- **修改 `buildGroups` 函数**：
  - single 类型也用 `outputItemKeySet.size` 计算 slotCost

### `process-ordersV4.js` 修改
- 修改 `processSingleBatch` 函数，让 single 类型也按 `upcDescList` 分组，*2、*3 分别生成 cover
- 修改 `normalizeCoverSkuLines` 函数，把 cover SKU 中的 UPC 也转成小写

### tianju-web-app 项目修改
- **同步修改分组逻辑**（`src/utils/index.js`）：
  - 修改 `extractAndGroupByUpcUnique` 函数，让 single 类型按 UPC 小写 + upcDescList 精确匹配分组

- **Dashboard UI 优化**（`src/pages/Dashboard.jsx`）：
  - 添加全局 loading spin 效果
  - 在 content-wrapper 外面包裹 Spin 组件

### 验证结果
- single 类型批次现在大部分都填满了 15 个不同 SKU
- 只有最后一个批次不满（这是正常的）
- 相同的 UPC 不会被切分到不同批次
- 不会出现数据丢失问题

## 2026-04-08

### 核心问题分析
发现了 UPC 大小写敏感匹配问题：
- 原始数据中订单 UPC 是 "Ody-Marshmallow"（大写 M）
- 拣货单 labelDesc 中 UPC 是 "Ody-marshmallow"（小写 m）
- 因为大小写不同导致匹配失败，拣货单缺失

### 问题根源定位
1. **`allocateLabelItemsByDemand` 函数大小写敏感**：
   - 第 351 行使用 `if (itemUpc !== upc)` 严格比较
   - 导致 "Ody-Marshmallow" 和 "Ody-marshmallow" 无法匹配
   - 影响范围：multiple 类型的 UPC 匹配

2. **库存 UPC 重复分析**：
   - 检查了 1856 个库存 UPC
   - 发现只有 1 组大小写不同的重复："Santal 33" vs "santal 33"
   - 其余 286 个重复都是完全相同的字符串重复

### `optimize-pick-batches.js` 修复

#### 1. multiple 类型 UPC 匹配修复
- **修改 `allocateLabelItemsByDemand` 函数**（第 351 行）：
  ```javascript
  // 修改前
  if (itemUpc !== upc) continue;
  
  // 修改后
  if (itemUpc.toLowerCase() !== upc.toLowerCase()) continue;
  ```
- 使用 `.toLowerCase()` 进行大小写不敏感比较
- 确保 "Ody-Marshmallow" 能正确匹配到 "Ody-marshmallow"

#### 2. 验证结果
- 修复前：self.json 中 multiple 类型有 61 个 UPC 找到匹配
- 修复后：self.json 中 multiple 类型有 72 个 UPC 找到匹配
- 新增找到的 UPC 包括："Ody-Marshmallow"、"luminous pink"、"Rhubarb" 等

#### 3. 潜在问题分析
- **single 类型**：通过 orderNo 关联，不受影响
- **multiple 类型**：已修复匹配问题
- **聚合 key 问题**：`buildAggregatedOutputItemKey` 函数中的 UPC 仍区分大小写
  - 可能导致同一库位、相同 UPC（只是大小写不同）不会被聚合
  - 需要进一步评估是否需要修复

### 两种类型的处理差异
| 类型 | UPC 匹配方式 | labelItems 获取方式 |
|------|-------------|-------------------|
| **single** | 不涉及 UPC 匹配 | 通过 orderNo 从 orderLabelMap 回收 |
| **multiple** | 大小写不敏感匹配（已修复） | 通过 UPC 从 labelPool 分配 |

## 2026-04-07

### 核心问题分析
发现了 pick 货位排序混乱问题：
- CC 货区后面紧跟着 HH 货区，从优化算法角度不应该发生
- single 类型中出现"2200 还没排完就塞了其他 zone"的情况

### 问题根源定位
1. **单个 UPC 组本身跨 zone 问题：
   - single 类型 284 个 group 中有 2 个（0.7%）包含多个 zone
   - multiple 类型 24 个 group 中有 4 个（16.7%）包含多个 zone
   - 这是仓库库存分布的原始数据问题

### `optimize-pick-batches.js` 重大改进

#### 1. 评分算法调整
- **modified `scoreCandidate` 函数**：
  - 从完全禁止不同货区前缀改为加大惩罚
  - sharedPrefixes 权重从 5000 增加到 10000
  - 新增无 sharedPrefixes 时的惩罚：如果没有共享货区前缀，扣减 10000 分
  - 这样在保证数量控制的前提下，尽量让同货区前缀的订单在一起

#### 2. single 类型优化策略重构
- **优化前问题**：使用启发式聚类算法，可能导致路径回退
- **优化后改进**：改用全局路径排序 + 箱容切分
- **新增 `compareSingleGroupsByPath` 函数**：
  - 优先按 dominantZone 排序
  - 再按 dominantShelf 排序
  - 最后按 anchorPosition 的 row/column/side 排序
- **重构 `optimizeSingleBatchesByPath` 函数**：
  - 撤销过于激进的货区前缀强制检查
  - 只按箱容分批，保持路径连续性
  - 确保从仓库头走到尾的最优路径

#### 3. multiple 类型启发式优化算法种子选择逻辑确认
- **种子选择策略**：优先选择库位靠前的 group
- **排序优先级**（`compareGroupsByAnchor` 函数）：
  1. dominantZone（zone 靠前
  2. anchorPosition.row（行靠前）
  3. anchorPosition.column（列靠前）
  4. anchorPosition.side（侧靠前）
  5. totalQty（件数多的优先）
  6. upcUnique（最后按 UPC 唯一标识）

### 两种算法对比
| 特性 | single（全局路径排序） | multiple（启发式聚类） |
|------|------------------------|------------------------|
| **适用场景** | group 之间独立，无绑定 | group 之间有组合绑定 |
| **排序方式** | 全局线性排序（类似蛇形路径） | 局部聚类（种子 + 最佳候选） |
| **路径连续性** | ✅ 最佳，从仓库头走到尾 | ⚠️ 一般，可能有回退 |
| **组合保持** | ❌ 不考虑 | ✅ 考虑 sharedShelves/sharedZones 等 |
| **实现复杂度** | 简单 | 复杂 |

### 新增辅助脚本
- `analyze-groups.js`：分析单个 group 是否包含多个 zone
- `verify-fix2.js`：验证修复效果的脚本

## 2026-03-29

### `optimize-pick-batches.js`
- 新增拣货批次优化脚本。
- 支持读取 `single` / `multiple` 两类数据结构。
- 保持 `upcUniqueList` 与 `orders` 的索引对应关系不变。
- 按每组最小 `upcUniqueList[i] + orders[i]` 作为重组单元。
- 优先按 `zoneName`、`shelfName` 进行聚合，尽量减少拣货员跨区移动。
- 保持每批最多 `15` 个 UPC 类别，可通过 `--box-size` 调整。
- `single` 类型下，组内订单按数量从大到小排序，例如 `*4` 在 `*1` 前。

### `process-ordersV4.js`
- 已集成 `optimize-pick-batches.js`。
- 执行 `node process-ordersV4.js xxx.json` 时，会先自动优化，再继续生成 PDF。
- 自动优化后的总 JSON 会直接覆盖原始输入文件。
- 新增 `--skip-optimize` 参数，用于跳过优化并保留原 JSON。
- 新增 `--box-size=15` 参数，用于控制优化阶段每批最大 UPC 数量。
- 处理前会输出优化统计信息，方便快速确认批次数、zone/shelf 切换次数。
- 修复 PDF 合成顺序：改为严格按 JSON 中订单遍历顺序合成，不再受文件名字典序影响。
- 下载的面单文件名新增组内序号前缀，进一步保证同组内顺序稳定。
- 延续 V4 已有的批次目录 + `u{upcIndex}` 文件名前缀策略，规避 Windows 大小写不敏感导致的重名覆盖问题。
